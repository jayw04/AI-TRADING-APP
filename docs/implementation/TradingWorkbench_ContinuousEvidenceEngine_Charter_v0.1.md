> **Purpose (frozen).** The Continuous Evidence Engine **does not decide whether a strategy is "good" or
> "bad." It decides whether live behavior remains consistent with the evidence that justified deployment.**
> It is continuous *model validation*, not a performance dashboard.

> *"Deployment is not the end of research. Deployment begins Continuous Evidence."*

---

## §0 — Why it exists (the third pillar)

The platform has moved from `Research → Paper Trading` to `Research → Live Paper → Continuous
Verification`. Discovery Lab validates a capability; Portfolio Engineering combines validated
capabilities; the CEE **verifies, in live operation, that realized behavior stays within the research
envelope that justified deployment.** Without it, "validated in 2026" silently decays into "assumed still
true" — the exact failure mode Evidence Engineering exists to prevent.

## §1 — Frozen design principles

These are non-negotiable and define the engine's character:

1. **Skeptical by default.** The default state is **Unknown / Insufficient Evidence**, never Pass or Fail.
   For the first weeks–months the engine should overwhelmingly report "insufficient" or "preliminary" —
   accumulating confidence slowly is the feature, not a limitation.
2. **Distributions, not point estimates.** The question is never "is Sharpe lower?" It is **"is the
   observed metric plausibly generated by the research distribution?"** — compared against the research
   *confidence interval* (which the platform already computes), not a single number.
3. **It observes; it never optimizes.** *(Evidence Principle #9.)* The CEE never auto-adjusts a strategy
   parameter or a portfolio weight. It surfaces evidence; humans decide. (Consistent with ADR 0035 —
   self-heal operations, never trading decisions.)
4. **Live observations accumulate evidence; they do not rewrite research.** *(Evidence Principle #8.)*
   A single bad day is *added to the evidence set*, not a trigger to change the strategy. Only sustained,
   statistically-meaningful drift reopens a research program.
5. **Operational drift ≠ investment drift.** The two are measured and reported on separate tracks and
   never mixed.

## §2 — The four evidence states (no automatic "Failure")

Every metric, per book, resolves to exactly one:

| State | Meaning |
|---|---|
| **Insufficient Evidence** | Too little live data to compare meaningfully (the default). |
| **Consistent** | Live behavior is within the expected evidence envelope. |
| **Watch** | Drift observed but not statistically meaningful. |
| **Investigate** | Persistent or statistically meaningful drift. |

There is deliberately **no "Failure"** state — the engine escalates to *Investigate*, and a human decides.

## §3 — Progressive confidence (confidence grows with live history)

The engine's confidence label is a pure function of accumulated live history — communicated explicitly so
no one over-reads early results:

| Live history | Confidence label |
|---|---|
| < 1 month | Insufficient Evidence |
| 1–3 months | Preliminary Observation |
| 3–6 months | Emerging Evidence |
| 6–12 months | Moderate Confidence |
| > 12 months | Mature Evidence |

## §4 — Operational drift vs investment drift (never mixed)

| Operational Drift (is the *machine* healthy?) | Investment Drift (is the *edge* behaving?) |
|---|---|
| missed schedules · stale data · execution failures · order rejects · broker issues | Sharpe · correlation · drawdown · turnover · exposure · win-rate · holdings overlap |

Operational drift is a *fix-the-system* signal (it already has partial coverage: the daily-report
watchdog, self-healing health-checks, CloudWatch alarms). Investment drift is an *is-the-research-still-
true* signal — the genuinely new capability. They share a report but never a verdict.

## §5 — How drift is measured (no hard thresholds initially)

For each investment metric the engine reports a **row, not a boolean**:

`Observed · Expected (research CI) · Difference · Confidence · Recommendation`

Example (early, honest):

> Research Sharpe **1.04** · Live estimate **0.96** · Confidence **Low** · Recommendation **Continue
> collecting evidence.**

A metric graduates from *Consistent* → *Watch* → *Investigate* only when the observed distribution
**separates** from the research CI (the same "excludes zero" logic used across the platform), *and* the
progressive-confidence clock has enough history to make the call. No fixed "Sharpe drops 20% → alert"
rule — those come later, if ever, and only once calibrated against real live data.

## §6 — Reuse map (build on what exists; don't rebuild)

- **Expected side (research envelope):** the per-program evidence packages already carry bootstrap CIs
  (`app/factor_data/evidence.py`: `paired_sharpe_diff_ci`, `ConfidenceResult`, `drawdown_profile`).
- **Observed side (live):** `EquitySnapshot` + the **Portfolio Analytics Engine** (`app/services/
  portfolio_analytics.py`, #322 — already computes live correlation / overlap / diversification) + the
  audit log.
- **Home:** likely `app/services/continuous_evidence.py` (or extend the CLAUDE.md-reserved
  `drift_detection.py`) — read-only, off the order path.
- **Output:** **Daily Report v3** surfaces per-book state + the Evidence Clock; escalation is *alert-only*
  with an audit trail (ADR 0035 discipline).

## §7 — The Evidence Clock (a first-class capability)

Every live book shows, always: **Days Live · Evidence Collected · Confidence · Last Review · Next
Review.** Example: *Momentum — 62 trading days · Confidence Low · Next review 30 days.* This is the
anti-overreaction device — it makes "we don't know yet" visible and legitimate.

## §8 — Pre-registered scope & phasing

- **Phase 1 — Envelope + Evidence Clock (read-only).** Persist each live book's research envelope (the
  expected CIs); compute the observed metrics from `EquitySnapshot`; emit the four-state + Evidence-Clock
  row per book into Daily Report v3. Overwhelmingly "Insufficient Evidence" at first — by design.
- **Phase 2 — Probabilistic drift.** Distribution-comparison per metric (observed vs research CI);
  Watch/Investigate escalation gated by the confidence clock; operational-vs-investment split.
- **Phase 3 — Continuous verification loop.** `Expected → Observed → Deviation → (human) Investigation`,
  audit-logged; feeds the reopen decision for a research program (never automatic).

## §9 — Non-goals (what it must never become)

- Not an adaptive optimizer (never changes params/weights).
- Not a performance dashboard (it measures *consistency with the justifying evidence*, not P&L).
- Not a fast alarm (skeptical, slow, honest > eager and noisy).

## §10 — Success criterion

The engine succeeds if, months from now, it can answer one question with statistical honesty for every
live book: **"Does reality still support the evidence that justified deploying this?"** — and if, in its
first weeks, it has the discipline to answer "we don't know yet" instead of pretending.

*Pre-registered 2026-07-02. Charter frozen; no engine built yet — this is the design, not the result.
The distinguishing characteristic vs conventional monitoring: it accumulates confidence slowly, reports
uncertainty honestly, and requires sustained evidence before changing any conclusion.*
