## Context

The New-Capabilities Build Spec v0.3 charts three additions to the platform's research side: (A) **alternative-data event ingestion** (add Quiver as a source behind the existing data-provider layer, feeding the point-in-time Event Store that INSIDER-001 established under ADR 0027); (B) **new event-driven research programs** siblings of INSIDER-001, MVP = **GOVCONTRACT-001** (Quiver government contracts → drift in small/mid-cap contractors); and (C) a human-facing **Daily Opportunity Report** that ranks and labels, never advises. The spec's discipline is explicit: *extend, do not rebuild* — no second event store, no parallel opportunity registry, no LLM in validation or the order path, and no customer-facing use of a dataset before its commercial license is confirmed in writing.

A Phase 0 confirmation pass against the repo (2026-07-05) validated most of the spec's anchors but surfaced **three divergences** where the spec's proposed shape does not match what is built:

1. **Schema gap.** The real Event Store (`app/altdata/events/store.py`, one generic `corporate_events` DuckDB table) has ~10 columns and a *single* PIT anchor, `filed_at`. The spec's canonical Event schema (§2.5) has ~24 fields — `available_time`, `resolved_security_id`, `raw_payload_hash`, `source_event_id`, `research_eligible`, `revision_time`, and more — none of which exist as columns.
2. **Security Master does not exist.** Only a SEC-scoped `CikMap` (from `company_tickers.json`) resolves identity today. There is no general issuer→ticker / ticker-change / delisting resolver — and Quiver government-contract recipients are frequently subsidiaries, divisions, or legal entities that do not directly match a traded parent.
3. **Opportunity Registry does not exist.** The spec's anchor claims Capability C "reads the Opportunity Registry" as an existing dependency. In fact the Registry is only **ADR-0029 Proposed**; the qualification/ranking/assignment logic is still fused inside the Range services (`range_insight.py`, `range_auto_select.py`) and persists only as an ephemeral `symbols_json`. (`api/v1/opportunities.py` is the dashboard aggregator — a different thing despite the name.)

The architect review resolved all three without abandoning the v0.3 approach. This ADR records those resolutions as coupled decisions, because they are not independent: they are the terms on which the *one* honest way to extend the INSIDER-001 event capability to paid alternative data is built. Per the platform invariant (a new external dependency requires an ADR), and per ADR 0027 §6 (a second, architecturally-different alt-data source is the trigger to generalize the alt-data framework), Quiver is exactly that second source — so this decision is recorded before ingestion code exists.

## Decision

**Extend the existing INSIDER-001 point-in-time Corporate-Event Store to paid alternative data (Quiver, government contracts first) under an explicit event-discovery governance regime, resolving the three Phase 0 divergences as follows.** No new event store, no new opportunity registry, no LLM in validation/ranking/order path.

### Governance principles (the regime every EAD program inherits)

1. **Discovery is not deployment.** An opportunity card is a research briefing with an evidence status; it is never a trade recommendation. Card labels are whitelisted (Watch · Research · Backtest Pending · Validated Pattern · Rejected Pattern) and the trade-advice vocabulary (Buy/Sell/Target Price/Expected Return/…) is forbidden (spec §4.3). External exposure is gated on both a licensing review (§2.4) and a compliance review.
2. **Every data source declares latency, licensing, and point-in-time availability before ingestion.** This is recorded in the Data Source Registry (below) and mirrored in the doc-level Data Capability Registry as **DCAP-007** (Quiver).
3. **Every generated hypothesis is pre-registered; every program clears a matched benchmark before promotion** — never raw event correlation. The matched-control recipe (~20 sector/mktcap/ADV/momentum-decile-±1 controls as-of `available_time`; excess-return 95% bootstrap CI must exclude zero) is mandatory (spec §3.2). This is the INSIDER-001 lesson — beta is not alpha — made a gate.
4. **Bounded testing.** Per-period hypothesis budget (1–3/week), Benjamini–Hochberg FDR control (q ≤ 0.10), one pre-registered primary hold window (20 trading days) with 5/10/60-day sensitivity checks, and mandatory held-out / walk-forward confirmation before any promotion.
5. **Previously rejected patterns are auto-checked before a research line reopens** (RNG-001, early INSIDER-001 must not resurface unflagged).
6. **LLM boundary (ADR 0006 v2, inherited).** An LLM may draft hypothesis prose and summarize opportunity cards. It must never decide validation, promotion, ranking weights, or enter the order path. The evidence gate stays deterministic.
7. **No customer-facing use of a dataset until its commercial, derived-signal, and cache rights are confirmed in writing.** Quiver Hobbyist and Trader both carry "No Commercial Use Rights" (verified 2026-07-05); only the contact-priced Commercial tier grants them. The MVP runs on **Hobbyist ($30/mo, month-to-month)** for internal R&D only.

### Architecture-reconciliation decisions (resolving the three Phase 0 divergences)

8. **Hybrid schema — extend `corporate_events`, do not hide governance fields in JSON.** Add the load-bearing PIT / identity / audit / licensing / eligibility fields as *real columns*; keep dataset-specific detail in the existing `payload` JSON. **No confidence/score column exists in the EAD v0 schema** — any score a normalizer emits stays inside `payload` and is structurally unreachable by eligibility or ranking until it is pre-registered and tested against outcome (the CONFIDENCE-UNINFORMATIVE precedent, re-evaluation trigger below). `available_time` becomes the canonical PIT anchor for all research/backtest/report reads; `filed_at` is retained (for Form-4 events `available_time = filed_at`, subject to the acceptance-vs-dissemination check in the migration note). `research_eligible` **defaults to `false`** and flips to `true` only when `available_time` is populated and validated *and* (for a public-company event) a confident security resolution exists — an unknown/unresolved mapping stays `false`. A `corporate_events_pit` compatibility view exposes `COALESCE(available_time, filed_at) AS pit_time` for legacy readers — **`ingested_at` is deliberately not a fallback** (ingestion can post-date availability by a lot on backfill, a look-ahead risk) — and new EAD programs never read `pit_time` at all: they require `research_eligible = true AND available_time IS NOT NULL`.
9. **Point-in-Time Security Master is a new formal capability, CAP-024** — not an extension of `CikMap`. Build a minimal v0 resolver: `resolve_security(issuer_name, ticker, cik, as_of) -> ResolutionResult` with the resolution hierarchy CIK → ticker-as-of-date → exact normalized-name → fuzzy name-match → else unresolved-with-reason. **The fuzzy tier resolves only if it clears a pre-registered similarity threshold; below it, the resolver returns `insufficient_confidence` (unresolved), never a resolved mapping.** Fuzzy matching is the classic source of silent bad mappings — the one thing this ADR refuses to tolerate — so the "no silent bad mapping" rule is enforced *in the fuzzy path itself*, not just asserted. For the MVP an *unresolved* mapping (with a typed reason) is acceptable; a *silent bad* mapping is not.
10. **The Daily Opportunity Report reads an audited adapter snapshot, not the Registry.** The Opportunity Registry is marked **Proposed (ADR-0029, not yet built)** in the spec. Capability C v0 reads the Event Store + SCAN-001 candidate outputs (+ `range_auto_select` outputs, GOVCONTRACT events) through an **`OpportunityInputAdapter`**. Because those live sources are *not* frozen/audited the way the real Registry will be, the adapter **snapshots and audit-logs its inputs at read time** — giving the v0 Report the same point-in-time repeatability the frozen Registry exists to provide, so no card is ever built on a candidate set that mutated intraday. When the formal Registry is built the adapter swaps its source without rewriting the report. The spec must state that `api/v1/opportunities.py` (dashboard aggregator) ≠ the Opportunity Registry (frozen, pre-open, audited Top-N machine contract).
11. **Order-path isolation is a CI invariant, not reviewer diligence.** Add a structural import guard — `scripts/check_altdata_order_path_isolation.sh`, modelled on `check_strategy_isolation.sh` — asserting that `app/altdata/quiver/**`, `app/altdata/security_master.py` (CAP-024), and `app/services/opportunity_report/**` import nothing from the order-path modules (`OrderRouter`, the risk engine, the broker adapters). The same argument the ADR makes for columns over JSON applies here: "a reviewer confirms the import boundary" is the convention that holds until the one PR nobody looks closely at. This becomes the fourteenth CI invariant.

### Registry-ID assignments (confirmed against the repo)

12. Quiver data source → **DCAP-007** (`Docs/implementation/TradingWorkbench_Data_Capability_Registry_v0.1.md`; current max DCAP-006). Point-in-Time Security Master → **CAP-024** (current max CAP-023). GOVCONTRACT-001 → new row in `app/research/programs.py` + the Research Program Registry. This ADR takes number **0037** (0034–0036 exist; the whitepaper cited only to 0033). These three registers are the ADR-0030 split in action — **CAP-024 is a Platform Capability** (infrastructure), **GOVCONTRACT-001 a Research Program** (evidence), **DCAP-007 a data input** — and each is onboarded through 0030's Capability Onboarding gate at the maturity level the spec assigns (Security Master and Quiver ingestion at L1; GOVCONTRACT-001 L0→L1). When ADR 0030's forthcoming Capability Manifest / versioned Capability Certificate land, CAP-024 carries one; this ADR does not pre-build that metadata layer.

## Rationale

- **Extend, not rebuild, because the store is already event-type-agnostic.** ADR 0027 deliberately put the PIT store at the *Corporate-Event* level with a free-form `event_type` string set by per-ingest `(EVENT_TYPE, SOURCE)` constants. A Quiver source is therefore a sibling of `app/altdata/sec/` writing a new `event_type` into the same `corporate_events` table — exactly the generalization ADR 0027 §6 named as the trigger for this framework ADR. A second store would fragment PIT correctness across two code paths and betray the spec's first non-goal.
- **Governance fields belong in columns, not JSON, because they are enforcement surfaces.** `available_time`, `research_eligible`, `raw_payload_hash`, `resolved_security_id`, and the licensing reference are read by PIT queries, eligibility filters, the audit chain, idempotency, and report gating. Buried in `payload`, each becomes a thing future code can silently bypass — the rule stops being load-bearing. The hybrid split (columns for what the platform *enforces*, JSON for what a dataset merely *carries*) is the honest boundary. Keeping `filed_at` and adding a compat view means no existing INSIDER-001 read breaks.
- **`available_time` over reusing `filed_at` directly, because not every alt-data event is a filing.** A government-contract award has an announcement time distinct from any filing; a 13F is a quarterly disclosure. Overloading `filed_at` to mean "the moment this became actionable" would be a lie for non-filing sources and would leak look-ahead the moment a dataset's event time ≠ its disclosure time. A dedicated canonical anchor makes the PIT rule explicit and uniform.
- **Security Master as a new capability, because the identity problem is genuinely different.** `CikMap` resolves SEC's own `company_tickers.json` — clean issuer identities keyed by CIK. Quiver arrives with contract recipients, subsidiaries, awardees, parent companies, and possibly stale names. That is a resolution problem `CikMap` was never built for; forcing it in would corrupt the one place the spec cannot afford corruption (a silent bad mapping produces a fabricated event study). A minimal v0 with typed unresolved reasons is the conservative default — coverage can grow later without an architecture change.
- **Not overbuilding the Security Master, because CRSP-grade corporate-action mastery is not the MVP.** The v0 contract handles the common cases and *labels* the rest unresolved; that is enough to run GOVCONTRACT-001 honestly and to measure mapping coverage against the kill floor. Full historical ticker-reuse / merger mastery is deferred until a program's evidence actually needs it.
- **Report reads an adapter, because blocking the EAD MVP on the unbuilt Registry would be the wrong dependency.** ADR 0029 is Proposed and gated on a separate trial. Making Capability C depend on it would stall the MVP behind unrelated Range work. The adapter preserves the eventual clean architecture (Report ← Registry) while letting v0 read what actually exists today — and prevents the drift of the Report growing its own selection logic.
- **Hobbyist, month-to-month, internal-only, because the pilot must stay disposable.** The point of the pilot is to *decide whether Quiver improves the platform*, not to adopt it. Government Contracts is a Hobbyist dataset, so Trader buys nothing for the MVP; month-to-month keeps the pilot killable under the §2.6a criteria before any annual prepay. And since neither Hobbyist nor Trader grants commercial rights, the Report stays internal until a written Commercial quote clears — a licensing fact, enforced by the `commercial_use_allowed` flag in the Data Source Registry.
- **Thresholds pre-registered as Phase-1 MVP gates, not permanent truths.** Magic numbers that are set after seeing results are not gates. Locking them now (with min/target/kill levels) makes each program genuinely pass/fail; labelling them MVP-scoped keeps them revisable when a program's evidence justifies it.

## Implementation notes

- **Schema migration (Event Store, `app/altdata/events/store.py`).** Add columns to `corporate_events`: `provider_dataset`, `source_event_id`, `available_time`, `revision_time`, `resolved_security_id`, `issuer_name_raw`, `ticker_raw`, `unresolved_reason`, `raw_payload_hash`, `data_source_id`, `research_eligible BOOLEAN DEFAULT FALSE`. Retain `filed_at`, `event_date`, `payload`. Add view `corporate_events_pit` (`COALESCE(available_time, filed_at) AS pit_time` — no `ingested_at` fallback). New PIT reads (`events_asof`) filter on `available_time`, not `filed_at`, for EAD event types.
- **The Form-4 `available_time` backfill is a separately-signed-off migration, not a bundled comment (review point 1).** It is the one step that can silently corrupt a working capability (INSIDER-001), so it carries its own gate:
  - **Acceptance-vs-dissemination check first.** `filed_at` is the SEC *acceptance* timestamp; confirm it is the public-availability moment for Form 4 (or characterize the acceptance→dissemination lag) *before* baking `available_time = filed_at` in as truth. If a material lag exists, `available_time` is derived from the dissemination time, not acceptance.
  - **Explicit default for unknown mappings.** A Form-4 row whose security resolution is *not* known is backfilled `research_eligible = false` (never unset/NULL-as-true). Eligibility is opt-in, per Decision 8.
  - **Reversible + verified.** The migration must be down-revisionable; the PR records a pre/post count of `research_eligible = true` rows and a spot-check proving **no historical INSIDER-001 read changes its result** (same event study output pre- and post-migration). A changed result blocks the migration.
- **Order-path isolation CI invariant (review point 2).** `scripts/check_altdata_order_path_isolation.sh` — a grep-style import guard over `app/altdata/quiver/**`, `app/altdata/security_master.py`, and `app/services/opportunity_report/**` asserting zero imports of `order_router` / `app/risk/**` / `app/brokers/**`. Wired into CI alongside the existing thirteen; this ADR authorizes the fourteenth invariant (CLAUDE.md invariant list to be updated on acceptance).
- **Quiver client** — sibling of `app/altdata/sec/`, e.g. `app/altdata/quiver/client.py`: bearer-token auth (key in `Settings`/`config.py` env-alias, read-only, off the order path, rides the ADR-0017 OS trust store), rate-limit aware. Government-contract normalizer `app/altdata/quiver/govcontracts.py` maps raw rows → `CorporateEvent` with `event_type="gov_contract_award"`, `source="quiver"`, both `event_time` and `available_time` populated.
- **Security Master (CAP-024)** — `app/altdata/security_master.py` (built here; the design's preferred `app/data/` proved to be gitignored, so this ADR's `app/altdata/` alternative was taken); `resolve_security(...) -> ResolutionResult { resolved_security_id, resolved_ticker, cik, confidence, method, unresolved_reason }`. Typed unresolved reasons: `ambiguous_name | no_public_security | subsidiary_unmapped | ticker_reused | insufficient_confidence`. May reuse `CikMap` as its CIK/ticker backbone. The fuzzy-name tier must return `insufficient_confidence` unless it clears the pre-registered similarity threshold (Decision 9) — no silent resolved mapping from fuzzy matching.
- **Data Source Registry** — a first-class entitlement/metadata table (`source_name, provider, datasets_enabled, license_type, commercial_use_allowed, redistribution_allowed, cache_allowed, derived_signal_allowed, refresh_frequency, known_latency, point_in_time_supported, contact_owner, renewal_date`), populated for Quiver *before* ingestion; its `commercial_use_allowed` / `derived_signal_allowed` flags gate what the Report may show. Mirrored at the doc level as DCAP-007.
- **Internal Data-Quality Report** (`app/services/data_quality/`) is built *before* any opportunity card: ingestion status · events by dataset · normalized · unresolved (with reason) · excluded-for-missing-`available_time` · duplicate count · mapping-failure rate · late-revision count · API failures · license status.
- **Opportunity Report** (`app/services/opportunity_report/` + `api/v1/opportunity_report.py` + a zero-dep frontend) reads the Event Store + SCAN candidates via `OpportunityInputAdapter`. The adapter **snapshots and audit-logs its input candidate set at read time** (Decision 10) so a report is reproducible from its snapshot and cannot reflect an intraday-mutated candidate set; the Report persists a daily audited snapshot on top of that; read-only, off the standard gate; zero trade recommendations.
- **Pre-registered Phase-1 MVP gates.** The **binding mapping gate** is a resolved *rate*, stated over the right denominator (review point 4): `confidently_resolved / public_company_events ≥ 85%` (proposed; confirm at Phase 0), with a **kill floor `< 70%` confidently resolved**. This is separate from the **completeness check** — `(confidently_resolved + explicitly_unresolved) = 100%`, i.e. every unresolved event carries a typed `unresolved_reason` — which is near-100% by construction and is *not* the constraint. (The two must not be conflated: a 60%-resolved run would fail the 85% gate even though it passes 100% completeness.) Plus: 100% of research-eligible events have validated `available_time`; 100% store `raw_payload_hash`; idempotent `event_id` across backfill/refresh; usable history ≥2yr (3+ preferred); ~20 matched controls (exclude/flag if <10); matching band sector/industry + mktcap/ADV/6m-momentum decile ±1; hypothesis budget 1–3/week; approval = 95% bootstrap CI on excess return excludes zero; BH FDR q ≤ 0.10; primary hold 20d + 5/10/60d sensitivity. **GOVCONTRACT-001 sample:** ≥100 eligible events after liquidity+mapping filters (300+ events / 50+ tickers preferred); kill if too few mapped/liquid/eligible events to test.
- **PIT correctness gate (§2.6).** A query as-of `T` returns only events with `available_time ≤ T`; for normal events `available_time ≤ ingestion_time`; any `available_time > ingestion_time` must be flagged corrected/backfilled with a `revision_time`, never silently accepted.
- **USAspending.gov cross-check (Phase 1 exit gate).** Before relying on Quiver, sample 100 government-contract events and reconcile against `api.usaspending.gov` (award date, recipient, agency, amount, ticker/company mapping, public-availability date) to validate Quiver's added value (public-company→ticker mapping) and protect the Evidence Engine from vendor normalization errors.
- **Off the order path — enforced by the new CI invariant above, not by review.** As with ADR 0027 the property is structural (the EAD packages import no order-path module); unlike ADR 0027, this ADR makes it load-bearing via `check_altdata_order_path_isolation.sh` rather than leaving it to reviewer diligence — consistent with the columns-not-conventions principle applied to the schema.

## Consequences

- **Positive:** the platform gains a *paid alternative-data* class through the same PIT-correct, read-only capability INSIDER-001 proved; the Event Store's governance rules become enforceable columns rather than JSON conventions; order-path isolation becomes a fourteenth CI invariant instead of a review convention; a reusable Security Master unblocks every future name/ticker/CIK-keyed source; the Report ships internally, reproducibly (audited adapter snapshot), without waiting on the unbuilt Opportunity Registry; the pilot is genuinely killable (Hobbyist month-to-month + pre-registered kill criteria).
- **Negative:** a schema migration on the live `corporate_events` store whose `available_time` backfill of historical Form-4 rows is risky enough to require its own sign-off (acceptance-vs-dissemination check + reversibility + a proof that no INSIDER-001 read changes result); a new paid external dependency to keep healthy (Quiver availability, rate limits, schema drift, and a "No Commercial Use" ceiling that blocks the external Report until a Commercial quote clears); a new resolver (CAP-024) whose v0 will *legitimately* leave some government-contract recipients unresolved, shrinking the tradable universe; the `OpportunityInputAdapter` (and its snapshot-audit) is interim scaffolding that must be retired when ADR-0029 lands (a known, deliberate debt); one more CI invariant to maintain.
- **Neutral:** the trading/risk/execution path is entirely untouched — additive on the research side, exactly like ADR 0018/0019/0027; the governance regime is discipline the platform already applies to strategies, extended to a new information class.

## Alternatives considered (not chosen)

- **Carry all new fields inside `payload` JSON (no column changes).** Rejected: the PIT/eligibility/audit/licensing fields are enforcement surfaces; in JSON they become bypassable and the rules stop being load-bearing. Reconsider only for genuinely dataset-specific detail (which does stay in `payload`).
- **Replace `filed_at` with `available_time`.** Rejected: breaks existing INSIDER-001 reads and erases the filing timestamp, which is legitimately distinct from disclosure availability for some sources. We keep both and add a compat view.
- **Extend `CikMap` into the Security Master.** Rejected: SEC-scoped clean-CIK resolution is a different problem from subsidiary/awardee/parent-entity mapping; overloading it risks silent bad mappings. A new minimal capability (CAP-024) with typed unresolved reasons is the honest model.
- **Block Capability C until the Opportunity Registry (ADR-0029) is built.** Rejected: stalls the EAD MVP behind unrelated Range-trial work; the adapter gives the same eventual architecture without the coupling.
- **A separate alt-data event store for Quiver.** Rejected outright — the spec's first non-goal and a direct violation of ADR 0027's event-type-agnostic design; fragments PIT correctness.
- **Buy Quiver Trader (or annual) for the MVP.** Rejected: Government Contracts is a Hobbyist dataset; Trader adds only "Later" datasets; month-to-month keeps the pilot disposable under the kill criteria. Upgrade only when a Trader-only dataset (Patents / App Ratings / 13F / insider cross-check / newsfeed) is actually reached.
- **Leave order-path isolation to reviewer diligence (as ADR 0027 did).** Rejected: it is the exact convention-not-mechanism failure mode this ADR argues against for the schema; a cheap grep-style CI check makes the property load-bearing. Reconsidered only if the check proves too brittle against legitimate refactors (unlikely for an import-boundary assertion).
- **Backfill `available_time = filed_at` for Form-4 rows inside the main migration, unverified.** Rejected: `filed_at` is an acceptance timestamp that may lag public dissemination, and an unverified backfill can silently change a working INSIDER-001 event study; the backfill is split into a separately-signed-off, reversible, result-verified migration.
- **Ingest 8–10 Quiver datasets in parallel.** Rejected (spec §2 MVP discipline): prove the full pipeline (available_time → PIT read → security-master resolution → data-quality report) on one explainable dataset, then fan out — mirroring how INSIDER-001 established the capability before generalizing.

## Re-evaluation triggers

- **Quiver pilot kill criteria fire (§2.6a):** <70% of contract rows map confidently; <2yr usable history; `available_time` unreconstructable; Quiver materially misses events found in USAspending.gov; duplicate/revision behaviour not idempotent; or too small a sample after liquidity filters — any one stops the pilot and reopens the "does Quiver improve the platform" question.
- **A commercial quote is obtained** (redistribution + derived-signal + cache rights confirmed in writing) — that is the trigger to revisit the internal-only ceiling and route the external Report through compliance.
- **The Opportunity Registry (ADR-0029) is built and Accepted** — the trigger to retire the `OpportunityInputAdapter` and point Capability C at the real Registry.
- **Security-Master unresolved fraction materially biases a program** — if typed-unresolved coverage is large enough to skew a government-contract (or later) event study, CAP-024 v0's minimal resolver becomes a first-class fix, not a known gap.
- **A confidence/score field is proposed on EAD events** — EAD v0 has *no* score column, so any score stays in `payload`, structurally out of eligibility and ranking (Decision 8). Promoting a score to a real, ranking-visible column requires pre-registering and testing it against outcome first (the CONFIDENCE-UNINFORMATIVE precedent) — that is a new decision, not a normalizer detail.
- **A third, architecturally-different alt-data source arrives** (news / macro / options as a *feed*, not a SCAN-001 config) — revisit whether the provider layer needs the abstract contract this ADR deliberately did not build.
